Once Kiro has finished generating the Current State Specs, Steering Documents, and Target Architecture, the next prompt should not ask it to write code. Instead, ask it to become the migration architect and produce a complete migration plan.

For an enterprise ETL migration like yours, I would use something like the following.

Kiro Migration Planning Prompt

Role

You are the Principal Enterprise Architect responsible for modernizing a mission-critical Spring Batch ETL system.

You have already analyzed the project and generated:

* Current State Specifications
* Technical Specifications
* Steering Documents
* Architecture Decision Records (ADR)
* Target Architecture
* Dependency Analysis
* Risk Analysis
* Performance Analysis

These documents are the single source of truth.

Do not re-analyze the project unless additional verification is required.

⸻

Objective

Create a complete migration plan for converting the existing single-threaded Spring Batch ETL application into a partition-based parallel processing architecture.

The migration must preserve 100% of existing business functionality.

This phase is planning only.

Do NOT generate implementation code.

⸻

Planning Principles

The migration plan must:

* Preserve business logic exactly.
* Preserve restartability.
* Preserve fault tolerance.
* Preserve transaction semantics.
* Preserve logging and auditing.
* Preserve file processing behaviour.
* Preserve validation behaviour.
* Preserve error handling.
* Preserve job parameters.
* Preserve ExecutionContext behaviour.
* Preserve output compatibility.

Only infrastructure should change.

⸻

Completion Rules

Do not generate implementation code.

Do not modify project files.

Do not create new classes.

Only produce planning documents.

If any uncertainty exists, document it instead of making assumptions.




The migration plan must be detailed enough that implementation can be performed one approved step at a time with minimal risk.


After Kiro produces these planning documents, pause and review them before starting any migration work. Once the plan is approved, you can use a much smaller implementation prompt for each individual step that references the migration plan rather than asking Kiro to rediscover the architecture each time.
